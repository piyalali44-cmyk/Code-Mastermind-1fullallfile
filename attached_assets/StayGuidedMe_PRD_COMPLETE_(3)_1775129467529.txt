USER CONTENT POLICY — LISTEN & VIEW ONLY (POCKET FM MODEL)
====================================================================================================

StayGuided Me follows a strict Pocket FM-style content model:

  USERS CAN   : Listen to audio content (Qur'an, Stories, Journey episodes)
  USERS CAN   : View all screens, categories, series, episode details
  USERS CAN   : Bookmark ayahs, add personal text notes, manage queue
  USERS CAN   : Download encrypted audio for offline playback (premium)
  USERS CAN   : Share deep links (URLs only — no audio files)

  USERS CANNOT: Upload any file (audio, image, video, document) to the platform
  USERS CANNOT: Edit or replace any content (episodes, covers, reciters)
  USERS CANNOT: Export any data as a file (PDF, CSV, ZIP) from within the app
  USERS CANNOT: Access any Supabase storage bucket with write permission

ALL CONTENT is uploaded and managed exclusively by Admins via the Admin Panel.
Storage buckets are write-protected from all app-side requests.
The app only reads from Supabase — it never writes to Storage.

----------------------------------------------------------------------------------------------------

    storage from the Flutter app — STOP. That is a violation of this policy.
    Admin Panel (Flutter Web) is the only interface allowed to write to storage.

A. Product Vision, Goals & Problem Statement
A.1 Vision Statement
StayGuided Me exists to make the complete intellectual and spiritual tradition of Islam — from the Qur'an to the life of the Prophet ﷺ and beyond — as accessible, beautifully designed, and engaging as any premium audio platform in the world, without ever compromising authenticity, depth, or reverence.
A.2 Problem Statement
+----------------------------------+-----------------------------------------------------------------------------------+--------------------------------------------------------------------------+
| Problem                          | Current Reality                                                                   | StayGuided Me Solution                                                   |
+----------------------------------+-----------------------------------------------------------------------------------+--------------------------------------------------------------------------+
| Fragmented experience            | Qur'an apps, story podcasts, and Islamic history content exist in separate apps   | All three combined in one beautifully designed platform                  |
| No chronological journey         | No platform guides users from Creation through all Prophets to today in one flow  | 'The Complete Story of Islam' — Noor's primary USP                       |
| No binge-listening model         | Islamic content is not structured for episodic, sequential listening              | Pocket FM-style episode model with autoplay and continue listening       |
| No dynamic context-aware content | No prayer times, Ramadan widgets, or regional Islamic event notices in audio apps | Admin-injected widgets: prayer times, countdowns, location-aware content |
| Poor UX quality                  | Most Islamic apps feel outdated, slow, and unpolished                             | Premium dark-theme audio-first design — Spotify quality                  |
| No admin control                 | Content teams need developer help to change banners or features                   | Full zero-code admin dashboard — everything without code                 |
| Unethical monetization           | Apps lock the Qur'an behind paywalls or show intrusive ads                        | Qur'an free forever — ethical freemium model                             |
+----------------------------------+-----------------------------------------------------------------------------------+--------------------------------------------------------------------------+
A.3 Core Design Principles
+-------------------------+-----------------------------------------------------------------------------------------+
| Principle               | What It Means                                                                           |
+-------------------------+-----------------------------------------------------------------------------------------+
| Audio-First             | Every screen designed around listening. Dark premium theme. Player always one tap away. |
| Qur'an Always Free      | Hardcoded. No admin toggle can change this. canPlaySurah() always returns true.         |
| Zero-Code Operations    | Admin changes any price, feature, layout, API, or banner without touching Flutter code. |
| Feature Flag Everything | Every feature has three independent switches: Enabled / Visible / Premium-Gated.        |
| Ethical Monetization    | Qur'an free forever. Premium = convenience + extras. Donation = voluntary sadaqah.      |
| API-Agnostic Qur'an     | Admin adds or switches any Qur'an API from the dashboard. No rebuild needed.            |
| Binge-Listening Model   | Series + episodes model. Autoplay next. Continue from last position. Like Pocket FM.    |
| Smart Sharing           | Shared links open directly inside the app. No audio files shared — only deep links.     |

H. Audio Architecture
H.1 SGAudioService — audio_service.dart
----------------------------------------------------------------------------------------------------
⚙  AI DEVELOPER NOTE — AUDIO ARCHITECTURE (BUILD IN THIS ORDER):
    Audio is the most critical system in the app. Build it carefully.

    Build order within this section:
    1. SGAudioService (H.1) — the core just_audio wrapper
    2. PlayerBloc (H.2) — state machine that drives all player UI
    3. H.3 Progress saving — the 10-second timer save loop
    4. QuranAudioResolver (H.4) — URL resolver (no hardcoded CDN URLs ever)

    Key rules:
    - SGAudioService is a singleton — initialise once in main.dart, inject via BLoC
    - Background play works via audio_service package's AudioHandler
    - Lock screen controls (Android MediaSession, iOS Now Playing) are auto-handled
      by the audio_service package when configured correctly
    - For encrypted offline files: ALWAYS call DownloadService.decryptToMemory()
      and pass the Uint8List to AudioSource.uri(dataFromBytes) — never write
      decrypted bytes to any file on disk
    - Progress is saved to Supabase every 10 seconds for logged-in users
    - For guests, progress is saved to Hive local storage (no Supabase call)
----------------------------------------------------------------------------------------------------
// lib/features/player/service/audio_service.dart
class SGAudioService {
final AudioPlayer _player = AudioPlayer();
Future<void> init() async {
await AudioSession.instance.then((s) =>
s.configure(const AudioSessionConfiguration.music()));
}
// Stream from URL — signed CDN URL or public Qur'an CDN
Future<void> playFromUrl(String url, {int startMs = 0}) async {
final tag = MediaItem(
id: contentId, title: title,
album: seriesName, artUri: Uri.parse(coverUrl),
);
await _player.setAudioSource(AudioSource.uri(Uri.parse(url), tag: tag));
if (startMs > 0) await _player.seek(Duration(milliseconds: startMs));
await _player.play();
}
// Play from encrypted local file — decrypts IN MEMORY, never writes plaintext to disk
Future<void> playFromEncryptedFile(String fileRef) async {
final bytes = await DownloadService.decryptToMemory(fileRef);
await _player.setAudioSource(
AudioSource.uri(Uri.dataFromBytes(bytes, mimeType: 'audio/mpeg')));
await _player.play();

Future<String> resolveUrl(int surah, {int? ayah}) async {

case 'qurancom_v4':       return await _fromQuranCom(surah, ayah);
case 'custom_cdn':        return _fillPattern(surah, ayah);
case 'custom_rest':       return await _fromCustomApi(surah, ayah);

}
}
String _fillPattern(int surah, int? ayah) {
final pattern = (ayah != null && source.urlPatternAyah != null)
? source.urlPatternAyah!
: source.urlPatternSurah!;
return pattern
.replaceAll('{surah_number}',         surah.toString())
.replaceAll('{surah_number_padded2}',  surah.toString().padLeft(2,'0'))
.replaceAll('{surah_number_padded3}',  surah.toString().padLeft(3,'0'))
.replaceAll('{ayah_number}',           (ayah ?? '').toString())
.replaceAll('{ayah_number_padded3}',   (ayah ?? '').toString().padLeft(3,'0'))
.replaceAll('{reciter_id}',            config.apiReciterId ?? '')
.replaceAll('{base_url}',              source.baseUrl)
.replaceAll('{bitrate}',               config.streamingBitrate.toString());
}
}
I. Secure Download System — AES-256 Encryption
⚑  Downloaded audio files are NEVER accessible from the device's file manager, iOS Files app, or any other app. Files are AES-256 encrypted on disk. Decryption happens IN MEMORY ONLY during playback. No plaintext bytes are ever written to any file.

----------------------------------------------------------------------------------------------------
⚙  AI DEVELOPER NOTE — DOWNLOAD SECURITY (NON-NEGOTIABLE RULES):
    This is a DRM (Digital Rights Management) system. Follow every rule exactly.

    The 4 rules you must never break:
    (1) NEVER write decrypted audio bytes to any file — decrypt to Uint8List in RAM only
    (2) NEVER use getExternalStorageDirectory() — use getApplicationDocumentsDirectory()
    (3) NEVER register a FileProvider or MIME type for .enc files
    (4) NEVER use Share.shareXFiles() for audio files — only deep links are shared

    Build the download flow in this exact order:
    Step 1 → _getSignedUrl()      : request signed URL from Edge Function
    Step 2 → _downloadBytes()     : download raw bytes via Dio
    Step 3 → _encryptAndSave()    : AES-256 encrypt → save .enc file → store key in Keystore
    Step 4 → decryptToMemory()    : decrypt → Uint8List → AudioSource.uri(dataFromBytes)

    On premium lapse: app checks download_grace_period_days setting. After grace period,
    delete all .enc files AND their keys from flutter_secure_storage.
----------------------------------------------------------------------------------------------------
I.1 Complete DownloadService Implementation
// lib/core/services/download_service.dart
class DownloadService {
// STEP 1: Request signed URL from Supabase Edge Function
Future<String> _getSignedUrl(String type, String id) async {
final res = await Supabase.instance.client.functions.invoke(
'get-signed-audio-url',
body: {'content_type': type, 'content_id': id},
);
if (res.data?['url'] == null) throw Exception('Signed URL not available');
return res.data!['url'] as String;
}
// STEP 2: Download file bytes via Dio with progress callback
Future<Uint8List> _downloadBytes(String url,
{void Function(int received, int total)? onProgress}) async {
final res = await dio.get<List<int>>(url,
options: Options(responseType: ResponseType.bytes),
onReceiveProgress: onProgress,
);
return Uint8List.fromList(res.data!);
}
// STEP 3: Generate unique AES-256 key + IV, encrypt, save to private dir
Future<String> _encryptAndSave(Uint8List bytes, String fileRef) async {
final key = Key.fromSecureRandom(32);  // 256-bit AES key
final iv  = IV.fromSecureRandom(16);   // 128-bit IV
final encrypter = Encrypter(AES(key));
final encrypted = encrypter.encryptBytes(bytes, iv: iv);
// Store key+IV in hardware-backed secure storage (NOT in files)
await const FlutterSecureStorage().write(
key: 'dl_key_$fileRef',
value: '${key.base64}:${iv.base64}',
);
// Save ENCRYPTED bytes to app-private directory
final dir = await getApplicationDocumentsDirectory();
final file = File('${dir.path}/downloads/$fileRef.enc');
await file.parent.create(recursive: true);
await file.writeAsBytes(encrypted.bytes);
return fileRef;
}
// STEP 4: Decrypt to MEMORY ONLY — never write plaintext to any file
static Future<Uint8List> decryptToMemory(String fileRef) async {
final keyData = await const FlutterSecureStorage().read(
key: 'dl_key_$fileRef');
if (keyData == null) throw Exception('Encryption key not found');
final parts     = keyData.split(':');
final key       = Key.fromBase64(parts[0]);
final iv        = IV.fromBase64(parts[1]);
final encrypter = Encrypter(AES(key));
final dir       = await getApplicationDocumentsDirectory();
final encBytes  = await File('${dir.path}/downloads/$fileRef.enc').readAsBytes();
return encrypter.decryptBytes(Encrypted(encBytes), iv: iv);
// Returns Uint8List → passed directly to AudioSource.uri(dataFromBytes)
// Decrypted bytes are NEVER written to any file
}
// Full orchestrated download flow
Future<void> downloadContent({
required String contentType,
required String contentId,
void Function(double progress)? onProgress,
}) async {
// Check access
if (!FeatureFlagService.downloadEnabled)
throw FeatureDisabledException('downloads');
if (FeatureFlagService.downloadRequiresLogin && !authBloc.isAuthenticated)
throw LoginRequiredException();
if (FeatureFlagService.downloadPremiumRequired && !authBloc.isPremium)
throw PremiumRequiredException();
final fileRef = '${contentType}_$contentId';
final url     = await _getSignedUrl(contentType, contentId);
final bytes   = await _downloadBytes(url,
onProgress: (recv, total) =>
onProgress?.call(total > 0 ? recv / total : 0));
await _encryptAndSave(bytes, fileRef);
// Register download in Supabase
await supabase.from('user_downloads').upsert({
'user_id': currentUser.id,
'content_type': contentType,
'content_id': contentId,
'local_file_ref': fileRef,
'file_size_bytes': bytes.length,
});
}
}
I.2 Security Rules Summary
+-------------------+---------------------------------------------------------------------------------------------------------------+
| Rule              | Implementation                                                                                                |
+-------------------+---------------------------------------------------------------------------------------------------------------+
| Storage location  | getApplicationDocumentsDirectory() — invisible to Android file manager and iOS Files app                      |
| Encryption        | AES-256 via encrypt package — unique random key + IV per file                                                 |
| Key storage       | flutter_secure_storage → Android Keystore / iOS Secure Enclave (hardware-backed)                              |
| Decryption        | In memory only. decryptToMemory() returns Uint8List → AudioSource.uri(dataFromBytes). Never writes plaintext. |
| No public URI     | No content:// URI. No FileProvider. No MIME type registration for audio files.                                |
| No share intent   | No SharePlus for audio files. Only deep links (URLs) are ever shared.                                         |
| Premium lapse     | On premium expiry: downloads remain accessible for download_grace_period_days. After grace: files deleted.    |
| Signed URL expiry | CDN signed URLs expire in 5 minutes — cannot be cached or shared outside the app.                             |

DECLARE v_coupon coupons%ROWTYPE;
BEGIN
SELECT * INTO v_coupon FROM coupons
WHERE UPPER(code) = UPPER(p_code) AND is_active = TRUE;
IF NOT FOUND THEN RETURN '{"valid":false,"error":"Invalid or expired coupon code"}'::JSONB; END IF;
IF v_coupon.expires_at IS NOT NULL AND v_coupon.expires_at < NOW() THEN
RETURN '{"valid":false,"error":"This coupon has expired"}'::JSONB; END IF;
IF v_coupon.max_uses IS NOT NULL AND v_coupon.uses_count >= v_coupon.max_uses THEN
RETURN '{"valid":false,"error":"This coupon has reached its usage limit"}'::JSONB; END IF;
IF EXISTS(SELECT 1 FROM coupon_redemptions WHERE coupon_id=v_coupon.id AND user_id=p_user_id) THEN
RETURN '{"valid":false,"error":"You have already used this coupon"}'::JSONB; END IF;
IF v_coupon.new_users_only AND EXISTS(SELECT 1 FROM subscriptions WHERE user_id=p_user_id) THEN
RETURN '{"valid":false,"error":"This code is for new subscribers only"}'::JSONB; END IF;
IF NOT (p_plan_type = ANY(v_coupon.applicable_plans)) THEN
RETURN '{"valid":false,"error":"This code is not valid for the selected plan"}'::JSONB; END IF;
RETURN jsonb_build_object(
'valid',true,'coupon_id',v_coupon.id,
'coupon_type',v_coupon.coupon_type,
'discount_value',v_coupon.discount_value,
'override_price',v_coupon.override_price);
END; $$ LANGUAGE plpgsql SECURITY DEFINER;

K. API Contracts
K.1 Key Supabase RPC Functions
----------------------------------------------------------------------------------------------------
⚙  AI DEVELOPER NOTE — API CONTRACTS (USE RPC, NOT DIRECT TABLE QUERIES):
    The app communicates with Supabase through these RPC functions — not through
    direct table SELECT queries on the client side (except for simple lookups).
    Always prefer calling the RPC function over writing a raw .from().select() query,
    because RPCs enforce business rules and security on the database side.

    Most important RPCs to implement early:
    - get_home_feed         : powers the entire Home screen — call on HomeBloc init
    - save_listening_progress : called every 10 seconds during playback (see H.3)
    - check_episode_access  : ALWAYS call before playing any premium episode
    - validate_coupon       : called when user taps 'Apply' on subscription screen
    - get_user_library      : powers all 8 tabs of the Library screen in one call

    Edge Functions (deploy via Supabase CLI or MCP):
    - get-signed-audio-url  : MUST be called before playing ANY premium episode
      Returns a 5-minute expiring signed URL. Never cache this URL beyond 4 minutes.
----------------------------------------------------------------------------------------------------
+-------------------------+------------------------------------------------------------------------+---------+-------+--------------------------------------------------------------------------------------+
| Function Name           | Parameters                                                             | Returns | Auth  | Purpose                                                                              |
+-------------------------+------------------------------------------------------------------------+---------+-------+--------------------------------------------------------------------------------------+
| get_home_feed           | lang TEXT, user_id UUID                                                | JSON    | No    | Returns all feed sections with content and active widgets                            |
| get_active_widgets      | zone TEXT, user_type TEXT, country TEXT                                | JSON[]  | No    | Returns highest-priority active widget per zone matching targeting                   |
| save_listening_progress | content_type, content_id, position, duration, completion, is_completed | void    | Yes   | Upserts playback position                                                            |
| get_user_library        | user_id UUID                                                           | JSON    | Yes   | Returns all library tabs: continue, favourites, downloads, bookmarks, notes, history |

| check_episode_access    | episode_id UUID, user_id UUID                                          | BOOLEAN | No    | Returns TRUE if user can access this episode                                         |

| get_journey_chapters    | user_id UUID                                                           | JSON[]  | No    | Returns all published chapters with user progress overlay                            |

K.2 Supabase Edge Function: get-signed-audio-url
// supabase/functions/get-signed-audio-url/index.ts
// Input:  { content_type: 'episode'|'surah', content_id: string }
// Output: { type: 'signed'|'public', url: string }
// Auth:   Required (JWT in Authorization header)
// Flow:
//   1. Verify JWT → get user
//   2. Call check_episode_access(content_id, user_id)
//   3. If access denied → return 403
//   4. Get audio storage path from episodes/quran tables
//   5. For Qur'an → return public CDN URL (no signing needed)
//   6. For private premium content → createSignedUrl(path, 300) — 5 min expiry
//   7. Return { type: 'signed', url: signedUrl }

L.1 Coupon Types
+---------------------+-------------------------------------------+-------------------------------------------------+
| Type                | How It Works                              | Example                                         |
+---------------------+-------------------------------------------+-------------------------------------------------+
| Percentage Discount | Reduces price by X%                       | RAMADAN30 → 30% off → $1.00 becomes $0.70/month |
| Fixed Amount Off    | Reduces by exact dollar amount            | SAVE50C → $0.50 off → $1.00 becomes $0.50       |

| Free Premium Period | N days completely free (no card needed)   | PARTNER60 → 60 free premium days                |
| Price Override      | Sets exact price regardless of plan price | HALFPRICE → always $0.50/month                  |

+---------------------+-------------------------------------------+-------------------------------------------------+
L.2 Coupon Validation Flow
User opens Subscription screen → taps 'Have a coupon code?' link
AnimatedSwitcher expands coupon input field
User types code (auto-capitalises) → taps 'Apply' button
App calls validate_coupon(code, user_id, plan_type) Supabase RPC
If valid: green banner '✓ RAMADAN30 applied — 30% off — $0.70/month' — price updates on screen
If invalid: red banner '✗ [specific error message]'
User taps 'Start Premium' → RevenueCat purchase with discounted price
On successful purchase → redeem_coupon() called → uses_count incremented

S4. Quran.com API — Complete Integration
StayGuided Me uses the Quran.com API v4 as the primary Qur'an data source. This provides: Arabic text, 90+ translations in 40+ languages, word-by-word with grammar, ayah timing for auto-scroll, tafsir, audio from 50+ reciters — all from one unified API.

----------------------------------------------------------------------------------------------------
⚙  AI DEVELOPER NOTE — QURAN.COM API (NO HARDCODING RULE):
    NEVER hardcode any Quran.com API URL, reciter ID, or translation ID directly
    in Flutter code. All API configuration is stored in the api_sources and
    reciter_api_config tables in Supabase and read at runtime.

    This means the admin can switch the entire Qur'an API source from the dashboard
    without any code change or app rebuild.

    Key implementation notes:
    - QuranApiClient reads its baseUrl from the api_sources table row where is_default=true
    - All reciter IDs come from the reciter_api_config table — never from code constants
    - Bengali translation default ID is 161 (Muhiuddin Khan) — set as default for
      users with language_pref = 'bn' in the user_settings or users table
    - Implement response caching: text/translation → 30 days, audio URLs → 7 days
    - Rate limit handling: if Quran.com API fails or returns 429, auto-fallback to
      the fallback_source_id configured in the api_sources row
    - Free tier: 1,000 requests/day. Apply at api.quran.com for production limits.

S4.4 Complete QuranApiService Implementation
// lib/features/quran/data/qurancom_service.dart
class QuranApiService {
final QuranApiClient _client;
final CacheBox _cache;  // Hive box for response caching
// ── Surah list ──────────────────────────────────────
Future<List<SurahModel>> getSurahList() async {
final cached = _cache.get('surahs');
if (cached != null) return SurahModel.fromJsonList(cached);
final res = await _client.dio.get('/chapters',
queryParameters: {'language': settingsService.language});
_cache.put('surahs', res.data['chapters'], ttl: Duration(days: 30));
return SurahModel.fromJsonList(res.data['chapters']);
}
// ── Ayahs with translation ──────────────────────────
Future<SurahDetailModel> getSurahWithVerses(int surahNum, {
String translationId = '131',  // Saheeh International default
bool wordByWord = false,
}) async {
final res = await _client.dio.get('/verses/by_chapter/\$surahNum',
queryParameters: {
'language': settingsService.language,
'words': wordByWord,
'translations': translationId,
'audio': currentReciterId,
'fields': 'text_uthmani,text_imlaei',
'word_fields': wordByWord ? 'text_uthmani,transliteration,translation' : null,
'per_page': 300,  // max — gets all verses in one call
}..removeWhere((k,v) => v == null),
);
return SurahDetailModel.fromJson(res.data);
}
// ── Ayah audio timing for auto-scroll ───────────────
Future<List<AyahTiming>> getAyahTimings(int surahNum, int reciterId) async {
final res = await _client.dio.get(
'/recitations/\$reciterId/by_chapter/\$surahNum',
queryParameters: {'per_page': 300},
);
return (res.data['audio_files'] as List)
.map((a) => AyahTiming(
ayahNumber: a['verse_number'],
audioUrl: a['audio_url'],
timestampMs: a['timestamp_from'],
)).toList();
}
// ── Tafsir ───────────────────────────────────────────
Future<List<TafsirVerse>> getTafsir(int surahNum, String tafsirId) async {
final res = await _client.dio.get(
'/tafsirs/\$tafsirId/by_chapter/\$surahNum',
queryParameters: {'language': settingsService.language},
);
return TafsirVerse.fromJsonList(res.data['tafsirs']);
}
// ── Search Qur'an ────────────────────────────────────
Future<QuranSearchResult> searchAyahs(String query) async {
final res = await _client.dio.get('/search',
queryParameters: {'q': query, 'size': 20, 'page': 1,
'language': settingsService.language},
);
return QuranSearchResult.fromJson(res.data);
}
// ── Available reciters list ──────────────────────────
Future<List<ReciterOption>> getAvailableReciters() async {
final res = await _client.dio.get('/resources/recitations',
queryParameters: {'language': 'en'},
);
return ReciterOption.fromJsonList(res.data['recitations']);
}
}

⚙  AI DEVELOPER NOTE — SURPRISE PLANS (HOW TO BUILD):
    Surprise Plans are a separate system from standard Monthly/Annual plans.
    They live in the surprise_plans table (see S5.2 for the schema).

    On the Flutter Subscription Screen:
    - At the TOP of the screen, above standard plans, show a horizontal scrollable row
      of active surprise plan cards (SurprisePlanCard widget)
    - Only show this section if: surprise_plans_enabled = true AND
      surprise_plans_on_subscription_screen = true (both from app_settings)
    - Filter active plans: is_active=true, active_from <= NOW, active_until >= NOW,
      purchases_count < max_purchases (or max_purchases IS NULL), target_segment matches user

    On purchase of a surprise plan:
    - Mobile: RevenueCat.purchaseProduct(plan.revenuecat_sku)
    - Web/Desktop: Stripe Checkout with plan.stripe_price_id
    - On success: INSERT into surprise_plan_purchases AND subscriptions tables
    - Auto-increment purchases_count — if it hits max_purchases, auto-set is_active=false

    Admin creates/manages all surprise plans from: Monetization → Surprise Plans
    No Flutter code changes needed when admin adds a new surprise plan.

S5.5 Purchase Flow — Surprise Plans
User taps 'Get This Deal' on a surprise plan card
App checks: is plan still active? purchases_count < max_purchases? user not already premium?
If all checks pass → proceed to payment
On mobile (iOS/Android): RevenueCat.purchaseProduct(plan.revenuecat_sku)
On web/desktop: Stripe Checkout → redirect to plan.stripe_price_id
On purchase success → INSERT into surprise_plan_purchases
→ INSERT into subscriptions (plan_type='admin_grant', duration = duration_days)
→ INCREMENT surprise_plans.purchases_count by 1
→ If purchases_count >= max_purchases → auto-set is_active = false
→ Send 'Welcome to Premium' push notification
→ Award XP: xp_subscription_start (from app_settings)

Section 1 — The Golden Rule
If a developer needs to be called to change any content, price, feature, layout, API, or rule — that is a failure of this admin panel. No code. No rebuild. No App Store wait.

----------------------------------------------------------------------------------------------------
⚙  AI DEVELOPER NOTE — THE GOLDEN RULE (DESIGN PHILOSOPHY):
    Every setting in the admin panel must write its value to the app_settings table
    in Supabase. The Flutter app re-reads app_settings on every launch and every
    30 minutes during a session. This is how admin changes reach users with no rebuild.

    When building any admin settings page, follow this pattern:
    1. Read current value: SELECT value FROM app_settings WHERE key = 'setting_key'
    2. Show the current value in the form field
    3. On save: UPDATE app_settings SET value = 'new_value' WHERE key = 'setting_key'
    4. Log the change to admin_activity_log (see Section 19 for logging pattern)
    5. Show success toast: 'Setting saved — app will update within 30 minutes'

    For critical settings (premium_enabled, maintenance_mode), also update
    Supabase Edge Config for near-instant propagation (under 60 seconds).

Section 8 — Widget Injection System
----------------------------------------------------------------------------------------------------
⚙  AI DEVELOPER NOTE — WIDGET INJECTION (DYNAMIC HOME FEED BLOCKS):
    The Widget Injection system allows the admin to insert any type of content block
    into any zone of the app without any code change. This is one of the most powerful
    features of the platform.

    How it works end-to-end:
    1. Admin creates a widget in the Admin Panel (Section 8 below)
    2. Widget is saved to the feed_widgets table in Supabase with is_active=true
    3. Flutter app calls get_active_widgets(zone, user_type, country) RPC on home load
    4. RPC returns the highest-priority active widget matching the targeting criteria
    5. FeedWidgetRenderer in Flutter routes to the correct renderer by widget_type string
    6. Widget renders inline in the home feed at the correct zone position

    The 12 injection zones:
    ZONE_GLOBAL_STICKY (pinned top bar), ZONE_HOME_TOP, ZONE_HOME_HERO,
    ZONE_FEED_SLOT_1, ZONE_FEED_SLOT_2, ZONE_FEED_SLOT_3, ZONE_FEED_INLINE,
    ZONE_QURAN_TOP, ZONE_LIBRARY_TOP, ZONE_PLAYER_BELOW,
    ZONE_PROFILE_PROMO, ZONE_SEARCH_BANNER

    Widget types to implement in Flutter (each has its own renderer):
    prayer_time, static_banner, countdown, hadith_block, custom_html, notice_bar,
    donation_cta, premium_upsell, new_series_card, ramadan_special, survey, iframe_embed

    Targeting: each widget can target by user_type (guest/free/premium/all),
    country_code, language, and platform. The Flutter evaluator filters widgets
    before rendering — users only see widgets relevant to them.

⚙  AI DEVELOPER NOTE — USER MANAGEMENT (PRIVACY & AUDIT RULES):
    The User Management section must follow strict privacy and safety rules.

    Privacy rules in the UI:
    - Email addresses in the user LIST must be masked: user@e***.com
    - Full email is only visible on the individual User Detail page
    - User IDs in logs should be shown as short prefix: a1b2c3d4... (first 8 chars)

    - Block User: requires reason text + confirmation dialog
    - Delete Account: requires typing 'DELETE' in a text field
    - Revoke Premium: requires reason text

    Admin actions that are REVERSIBLE can be done with one confirmation:
    - Grant Premium, Extend Premium, Unblock User, Reset Password, Send Push

    EVERY admin action on a user must be logged to admin_activity_log with:
    - admin_id (who did it), action (what was done), entity_type='user',
      entity_id (the user's ID), details (JSONB with before/after values), ip_address

    Grant Premium special rule (RULE-004 from P.1):
    When admin grants premium manually (is_manual=true), this subscription must NEVER
    be cancelled, modified, or expired by RevenueCat webhooks. Always check is_manual
    before processing any automatic subscription status change.

12.5 DELETE Flow — Path B: Coupon Has Redemptions
Admin clicks DELETE → system finds coupon_redemptions count > 0 (e.g., 47 redemptions)
Warning dialog opens (amber border, different from standard confirmation):
Title: 'Coupon RAMADAN30 has been used 47 times'
Body: 'Deleting will NOT affect users who already used it — their subscriptions stay active at the discounted price. However the code will stop working for anyone who tries to use it in future.'
Three buttons: [Cancel]   [Pause Instead — Recommended]   [Delete Anyway] (amber button)

If admin clicks Delete Anyway: SECOND confirmation dialog appears:
Title: 'Final confirmation — type the coupon code to confirm deletion'
Text input field: admin must type 'RAMADAN30' exactly — Delete button stays grey until text matches
Admin types exact code → Delete button turns red → admin clicks → coupon row deleted
Success toast: 'Coupon RAMADAN30 deleted. 47 past redemption records preserved for accounting.'
⚑  Deleting a coupon NEVER revokes discounts already given. If a user used RAMADAN30 for 30% off, their subscription continues at the discounted price forever. Past coupon_redemptions rows are ALWAYS preserved — they are never deleted, even when the parent coupon is.

⚙  AI DEVELOPER NOTE — REFERRAL SYSTEM (HOW THE REWARD FLOW WORKS):
    The referral system works through a combination of deep links + database triggers.

    Full referral flow:
    1. User A goes to ReferralScreen → sees their unique referral link:
       https://stayguided.me/join?ref=ABC123 (referral_code from users table)
    2. User A shares this link
    3. New User B clicks the link → App Store → installs app → opens app
    4. Deep link handler reads ?ref=ABC123 → stores referral_code in SharedPrefs
    5. User B signs up → INSERT users row with referred_by = User A's UUID
    6. User B subscribes → the referral-reward Edge Function fires (database webhook)
    7. Edge Function checks: is referral_reward_premium_required = true?
       If yes and User B subscribed → grant User A + User B N days premium each
       If no → grant reward on signup only
    8. INSERT into referral_rewards table recording both users' rewards

    Admin controls all reward amounts from Settings → Referral System:
    referral_reward_days_referrer, referral_reward_days_referee, referral_max_rewards

    The referral_code for each user is auto-generated on signup:
    Generate a random 6-character alphanumeric code → check uniqueness → save to users.referral_code
